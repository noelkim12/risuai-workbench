# AGENTS.md

이 저장소에서 작업하는 AI 코딩 에이전트 공통 작업 규칙입니다.

## Mandatory TODO Workflow

1. 작업 시작 전에 루트 `TODO.md`를 읽고 현재 남은 항목을 파악한다.
2. 잔여 작업에 대해 TODO 목록을 지속적으로 업데이트한다.
3. 작업이 끝나면 루트 `FIN.md`를 반드시 최신 상태로 갱신한다.
4. 최종 보고에는 다음 두 가지를 반드시 포함한다.
   - 이번 작업으로 완료된 TODO 항목
   - 아직 남아 있는 TODO(remain tasks) 리마인드
5. `TODO.md`의 `### Remaining`은 반드시 기능 단위 섹션(`#### <SECTION>`)으로 유지한다.

## Completion Rule

- "완료"라고 말하기 전에 `TODO.md`의 관련 항목 상태가 실제 코드/문서 상태와 일치해야 한다.
- 남은 작업이 하나라도 있으면 최종 응답에서 명시적으로 알려준다.
- remain tasks를 리마인드할 때는 항목뿐 아니라 해당 `#### <SECTION>` 이름도 함께 안내한다.

## Git Safety (Hard Block)

- **절대로 `git reset`을 실행하지 않는다.** `--soft`, `--mixed`, `--hard`, commit hash 대상 reset, pathspec reset을 모두 포함해 **전면 금지**한다.
- 에이전트는 어떤 이유로도 작업 트리/인덱스/히스토리를 `git reset`으로 되돌리거나 재작성해서는 안 된다.
- 사용자가 명시적으로 요청하더라도, `git reset`은 기본적으로 수행 금지로 간주하고 더 안전한 대안을 먼저 제안한다.
- 커밋/스테이징/작업물 복구가 필요하면 `git status`, `git diff`, `git log`, `git reflog` 같은 **읽기 전용 조사 명령**으로만 먼저 파악한다.
- 파일 복원이나 변경 취소가 정말 필요하면, 에이전트가 독단적으로 파괴적 git 명령을 실행하지 말고 현재 상태를 보존하는 방향으로 별도 패치/백업/수동 복사 방법을 우선 사용한다.
- 이 규칙 위반은 중대한 사고로 간주한다. 에이전트는 **작업물 유실 가능성이 있는 git 명령을 선제적으로 차단**해야 한다.

## File Deletion Safety (Hard Block)

- **절대로 임의의 `rm`, `unlink`, delete, cleanup 명령이나 파일 삭제성 편집을 실행하지 않는다.** 일반 파일 삭제도 작업물 유실로 이어질 수 있으므로 git 명령과 같은 수준으로 차단한다.
- untracked 파일, 사용자가 만든 파일, 현재 작업트리의 생성물은 사용자 확인 없이 삭제하지 않는다. git이 추적하지 않는다는 이유는 삭제 허가가 아니다.
- 문서, 계획서, `TODO.md`, `FIN.md`, notepad, 생성된 리포트와 같은 작업 산출물은 현재 작업 범위 밖이거나 scope creep처럼 보여도 자동 정리하지 않는다.
- "out of scope", "scope creep", "cleanup" 판단은 파일 삭제 권한이 아니다. 삭제가 필요해 보이면 먼저 읽기 전용 조사만 수행하고, 삭제 후보 경로와 이유를 보고한 뒤 사용자에게 명시적 확인을 받아야 한다.
- 정리가 애매하면 삭제보다 격리, 백업, 그대로 두기를 우선한다. 에이전트 판단만으로 파일을 제거하지 않는다.
- subagent나 reviewer에게 검토를 맡길 때는 파일 제거, 복원, 정리 없이 **read-only 조사만 수행**하라고 명시한다.
- 이 규칙 위반은 작업물 유실과 동등한 중대한 사고로 간주한다. 에이전트는 모든 파일 삭제 가능 작업을 선제적으로 차단해야 한다.

## TONE

- 존댓말을 기본으로 쓴다.
- "~에요", "~거에요?", "~해줘요", "~있어요?", "~인 거 아니에요?" 같은 부드러운 어미를 자연스럽게 활용한다.
- 따뜻하고 친근한 분위기를 유지하되, 과하게 오글거리거나 작위적이지 않게 자연스럽게 표현한다.
- 이모지는 사용하지 않는다.
- 응답의 첫 문장은 말줄임표("..") 없이 깔끔하게 시작한다.

## Comment / JSDoc Rule

- 함수 주석은 기본적으로 **JSDoc 스타일(`/** ... */`)**로 작성한다.
- 최소한 다음 3가지는 반드시 포함한다.
  - 함수명
  - 함수 설명
  - 각 파라미터 설명 (`@param`)
- 반환값이 있는 함수는 가능하면 `@returns`도 함께 작성한다. 공개 함수, 공용 헬퍼, 테스트 fixture 유틸은 사실상 필수로 본다.
- 파일의 역할이 바로 드러나지 않거나, 공용 모듈/설정/fixture처럼 진입 설명이 중요한 파일은 **파일 최상단 JSDoc 개요 주석**을 둔다.
- 파일 개요 주석에는 최소한 다음 2가지를 포함한다.
  - 파일의 간략한 설명
  - `@file` 경로 또는 식별자
- 파일 개요 주석은 import 위, 파일의 가장 첫 부분에 둔다.
- 파일 개요 JSDoc 기본 형식은 아래를 따른다.

```ts
/**
 * 파일의 간략한 설명
 * @file 디렉토리/파일명
 */
```

- `@file` 값은 보통 `packages/foo/src/bar.ts`처럼 **저장소 기준 경로**를 적는다.
- 파일 설명은 "이 파일이 무엇을 모아두는지", "어디에서 쓰이는지"를 한 줄로 먼저 드러낸다.
  - 예: `cbs-lsp 테스트에서 공용으로 쓰는 fixture corpus 모음.`
  - 예: `문서 경로를 CBS artifact 타입으로 분류하는 유틸 모음.`
- 모든 파일에 기계적으로 붙이지는 않는다. 역할이 자명한 짧은 파일보다, **진입 비용이 있는 파일**에 우선 적용한다.
- JSDoc 기본 형식은 아래 순서를 따른다.

```ts
/**
 * 함수명.
 * 함수가 하는 일을 짧고 바로 이해되게 설명.
 *
 * @param paramName - 파라미터 용도 설명
 * @returns 반환값 설명
 */
```

- 설명 문체는 문서체 "~합니다"보다 **짧고 직접적인 전문적인 어조**를 우선한다.
  - 예: `createFixtureRequest 함수.`
  - 예: `fixture 한 건을 테스트 요청 형태로 변환함.`
- 함수 설명은 구현을 줄줄이 풀어쓰기보다 **역할과 용도**를 먼저 적는다.
- `@param` 설명은 타입 반복이 아니라 **그 값이 왜 필요한지**를 적는다.
  - 좋은 예: `@param area - 조회할 red test matrix 영역 이름`
  - 나쁜 예: `@param area - FixtureMatrixArea 타입 값`
- 주석은 길게 쓰기보다 **2~3줄 요약 + 필요한 태그만** 유지한다. 테스트 코드라고 해서 무조건 생략하지는 않지만, 이름만으로 충분히 읽히는 사소한 로컬 헬퍼에는 과한 주석을 달지 않는다.
- 인라인 주석(`// ...`)은 보조 설명 용도로만 쓰고, 함수 자체의 계약/입력/출력 설명은 JSDoc에 모은다.
- 새 함수 주석을 추가하거나 기존 주석을 수정할 때는, 주변 파일 스타일과 길이를 먼저 확인하고 그 파일의 밀도에 맞춘다.
